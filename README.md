# RAG-система по лору Warhammer 40,000

RAG-система по Warhammer 40k с гибридной архитектурой.

### Проблема
Лор Warhammer 40k содержит:
- сложные причинно-следственные связи между событиями
- множество сущностей (фракции, персонажи, кампании)
- информацию, распределённую по разным источникам

Классический RAG плохо справляется с multi-hop вопросами и теряет связи между сущностями.

### Решение
Гибридная архитектура:
- классический RAG (retrieval + reranker)
- графовый retrieval (LightRAG) для связей между сущностями

Достижения:
- faithfulness: 0.92
- answer_relevance: 0.92
- context_relevance: 0.88
  
### Примеры запросов
Пример сложного запроса:
```
Почему Расколотые легионы (Железные Руки, Гвардия Ворона, Саламандры) продолжали действовать как координированная сила после Истваана V, и как их вмешательство в инцидент на Тассосе повлияло на кампанию Льва Эль'Джонсона на юге?
```

Пример простого запроса:
```
Как Чернокаменные крепости связаны с Абаддоном?
```

## Что внутри
- **Фреймворк**: LangChain
- **Векторная БД**: Qdrant 
- **Embeddings**: TEI (Text Embeddings Inference) — Qwen3-Embedding-0.6B
- **Reranker**: vLLM - bge-reranker-v2-m3
- **LLM**: GigaChat Pro / GigaChat Lite (через API Sber)
- **API**: FastAPI
- **Инфраструктура**:
  - Всё в Docker + Docker Compose
  - Tracing & Observability — **Langfuse**
- **Оркестрация запросов**:
  - Умный роутинг: простые факты → классический RAG, сложные связи → **LightRAG** (графовый)

## LightRAG
Графовый retrieval для multi-hop вопросов.
Используется, когда ответ требует:
- связей между фракциями
- исторических цепочек событий
- причинно-следственных зависимостей

## Метрики и эксперименты
### Оценка би-енкодера
Первичный выбор моделей производился на [MTEB](https://huggingface.co/spaces/mteb/leaderboard). Выбранные модели прогонялись на тестовом наборе, а также оценивались ембеддинги на двумерной плоскости.
Эксперименты: ```experiments/test_models.ipynb```

### Оценка ретривера
Для оценки качества ретривера мы собрали собственный датасет из 60 вопросов.
Каждый вопрос был размечен вручную:
* релевантные статьи (article_title)
* ключевые цитаты, которые должны находиться в retrieved чанках

Таким образом, разметка покрывает как документный уровень, так и уровень конкретных текстовых совпадений.
Использование реранкера показало рост метрик.
```
════════════════════════════════════════════════════════════════════════════════════════════════════
СРАВНЕНИЕ @5                 | Base       | Rerank     | Delta
----------------------------------------------------------------------------------------------------
citation_recall@5            | 0.673      | 0.817      |     +0.144
citation_precision@5         | 0.156      | 0.190      |     +0.034
```
Конечные значение метрик предствалены в таблице.

### Оценка генерации
Для оценки качества финальных ответов используется подход LLM-as-a-Judge: отдельная языковая модель выступает в роли эксперта и валидирует ответы RAG-системы.

Модель возвращает структурированную оценку по трём ключевым критериям:
* **Faithfulness (0–1)**
Насколько ответ соответствует контексту
→ штрафует галлюцинации
* **Answer Relevance (0–1)**
Насколько ответ действительно отвечает на вопрос
* **Context Relevance (0–1)**
Насколько retrieved контекст был полезен для ответа
* **Critique**
Текстовое объяснение оценки

```
════════════════════════════════════════════════════════════════════════════════
Метрика                      Mean       Std       
--------------------------------------------------------------------------------
faithfulness                 0.921      0.231     
answer_relevance             0.926      0.199     
context_relevance            0.882      0.256
```
